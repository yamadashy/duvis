# duvis Project Structure and Overview

## Project Overview

duvis (Disk Usage Visualizer) is a disk usage analysis CLI tool designed for both AI agents and humans. It provides structured JSON output for AI consumption and human-readable tree/visual output for direct use. Built in Rust for performance and safety.

## Directory Structure

```
duvis/
├── src/
│   ├── main.rs       # Entry point (thin - delegates to lib)
│   ├── lib.rs         # Library public API
│   ├── cli.rs         # CLI argument parsing (clap)
│   ├── entry.rs       # Core data types (Entry, SortOrder)
│   ├── format.rs      # Shared formatting utilities
│   ├── scanner.rs     # Directory scanning logic
│   ├── ui.rs          # axum server that hosts the browser UI
│   ├── ui/
│   │   └── index.html # Built single-file React UI (generated by `cd ui && npm run build`)
│   └── output/        # Output renderers
│       ├── mod.rs     # OutputConfig and render dispatch
│       ├── json.rs    # JSON output
│       └── tree.rs    # Tree output
├── ui/                # React + TypeScript source for src/ui/index.html
│   ├── src/
│   ├── vite.config.ts
│   └── package.json
└── tests/             # Integration tests
```

## Common tasks

Day-to-day commands are bundled in `justfile`. Install [just](https://just.systems)
(`brew install just` or `cargo install just`) and run `just` to see the list.

```
just dev        # cargo run --release -- . --ui
just build      # cargo build --release
just test       # cargo test --lib
just lint       # fmt-check + clippy + ui typecheck (matches CI)
just check      # lint + test
just fmt        # auto-format Rust + TypeScript
just ui-dev     # vite dev server (UI iteration without Rust)
just package    # list files that would ship to crates.io
just publish    # cargo publish + git tag + GitHub release
just clean      # nuke build artifacts (Rust + UI)
```

## UI workflow

The browser UI shipped via `duvis --ui` lives in `ui/` as a small React +
TypeScript app built with Vite. It's bundled into a single self-contained
HTML file by `vite-plugin-singlefile`. The Rust crate embeds that file with
`include_str!` at compile time.

`build.rs` regenerates `src/ui/index.html` from `ui/src/` automatically on
every `cargo build` (via `npm --prefix ui run build`). The first build also
runs `npm ci` if `ui/node_modules` is missing. So as a developer you just
need Node.js installed; no manual `npm run build` step.

`src/ui/index.html` is gitignored — only the source under `ui/src/` is
checked in. The crates.io tarball still ships the file via the `include`
list in `Cargo.toml`, so `cargo install duvis` works without Node (build.rs
detects the absence of `ui/` and reuses the bundled file).

## Architecture Principles

- **main.rs stays thin**: CLI parsing and orchestration only. All logic lives in lib.rs and its modules.
- **lib.rs as public API**: Enables use as a library (for MCP server, tests, etc.).
- **Core types in entry.rs**: `Entry` and related types are the single source of truth for data structures.
- **Scanner is pure data collection**: No formatting or output logic in scanner.
- **Output modules are pure rendering**: No scanning or data mutation in output.

# Coding Guidelines

- The assistant shall follow idiomatic Rust style and conventions.
- The assistant shall use `cargo fmt` formatting.
- The assistant shall use `cargo clippy` for linting.
- If a file exceeds 250 lines, the assistant shall consider splitting it into smaller modules.
- Where non-obvious logic exists, the assistant shall add comments in English to clarify.
- When implementing new features, the assistant shall provide corresponding tests.
- When implementation is complete, the assistant shall verify changes by running:
  ```bash
  cargo fmt --check  # Ensure code style compliance
  cargo clippy        # Lint checks
  cargo test          # Verify all tests pass
  cargo build         # Ensure it compiles
  ```

## Error Handling

- Use `anyhow::Result` for application-level errors.
- Use `thiserror` for library-level error types if needed.
- Silently skip permission-denied errors during scanning (read-only tool, graceful degradation).

## Commit Messages

- The assistant shall follow the [Conventional Commits](https://www.conventionalcommits.org/) specification for all commit messages.
- The assistant shall write commit messages in English.
- The assistant shall include a scope in commit messages.
- The assistant shall use the format: `type(scope): Description`
  ```
  # Examples:
  feat(cli): Add --exclude flag for filtering directories
  feat(scanner): Add parallel directory traversal with rayon
  feat(output): Add size bar visualization in tree output
  fix(scanner): Handle symlink loops gracefully
  refactor(entry): Extract file classification into separate module
  docs(readme): Add installation instructions
  test(scanner): Add tests for permission denied handling
  chore(deps): Update clap to v4.6
  ```
- The assistant shall use types: feat, fix, docs, style, refactor, test, chore, etc.
- The assistant shall use scope to indicate the affected module (cli, scanner, output, entry, format, etc.).
- The assistant shall write description in clear, concise present tense starting with a capital letter.
